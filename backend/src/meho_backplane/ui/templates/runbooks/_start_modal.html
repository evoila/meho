{#- SPDX-License-Identifier: Apache-2.0 Copyright (c) 2026 evoila Group HTMX-loaded start-run modal for the runbook runs surface. Loaded into ``#runbook-runs-modal-container`` on the runs page by the "+ Start run" button's ``hx-get`` to ``/ui/runbooks/runs/start``. The shared app-shell modal controller (``app/modal-dialogs.js``) opens the inserted ```` via ``showModal()`` on ``htmx:afterSwap`` so the operator sees the modal without an extra click. Initiative #1837 (G10.11 Runbook runs UI), Task #1884 (T1). Submit flow: the form ``hx-post``s to ``POST /ui/runbooks/runs``. * SUCCESS -> 204 with ``HX-Redirect: /ui/runbooks/runs/{run_id}`` -- HTMX navigates straight into the T2 run driver (the empty 204 body makes the innerHTML swap a no-op; the redirect fires first). * TYPED START ERROR (deprecated template / not found / missing ``${run.params.X}``) -> a 200 re-render of THIS modal carrying an inline ``alert-error`` (``error_message`` set). Because the form targets ``#runbook-runs-modal-container`` with ``hx-swap="innerHTML"``, the re-rendered modal replaces this one in place and the controller re-opens it (a fresh, not-yet-open ```` node), so the operator sees the message -- e.g. exactly which run.params key to supply -- and can correct + resubmit without losing their place. CSRF posture: the form declares its own ``hx-headers`` echo of the token THIS render minted; the route sets the matching ``meho_csrf`` cookie on the same response, so the double-submit pair always lines up (#1693). Relying on the page-level token is not enough -- the modal render rotates the cookie, so an inherited header would carry the stale list-render token while the cookie carried the fresh one and the chassis CSRF middleware would 403 the start POST (``value_mismatch``). On the typed-error re-render the token is re-minted + cookie refreshed so the retry matches. -#}
