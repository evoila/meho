Shows the operator-relevant request detail (op_id / connector_id / proposed_effect / requester principal_sub / created_at) and the Approve + Deny actions. Self-approval invariant (#1401) The **Approve** button is disabled (read-only) when the operator is the requester and the audited break-glass is off (``self_approval_blocked``); its ``aria-label`` names the ``APPROVAL_ALLOW_SELF_APPROVAL`` flag so the operator learns the lever. **Deny** stays enabled (withdrawing your own request is not an escalation). The route minted a fresh token for THIS render and re-set the matching ``meho_csrf`` cookie on the same response, so the double-submit pair lines up after the modal swap rotated it. Decided rows render read-only (#1827) When the request has already been decided (``status != "pending"``) the modal shows a decision banner ("Approved by X at T" / "Rejected by X at T" / "Expired at T") and the Approve/Deny actions are NOT rendered -- there is nothing to act on. Only a pending row offers the decision forms. This is what lets the history view's "View" button reuse the same modal for a closed request. An expired row has a null ``decided_at`` (it was closed by timeout, not a decision), so the "at T" falls back to ``expires_at`` for that branch. Confirm gate (#2446) A pending row carries a consequence banner (``data-approval-gate="confirm"``) whose severity follows the envelope ``safety_level`` (#1855) -- error for ``dangerous``, warning otherwise -- naming what each decision does: Approve dispatches the parked write immediately (no un-approve), Deny is terminal (re-file). Both decisions are two-phase: the first click on an armer (``type="button"``, Alpine local ``armed`` state) only swaps in a Confirm pair; only the Confirm button posts. This mirrors the runbook lifecycle confirm mold (``runbooks/_detail_actions.html``) and the ops Run gate (#1881). A blocked self-approval Approve (#1401) never arms.
