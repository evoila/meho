Swapped into ``#meho-approvals-modal`` by the bell / the panel's "Review" button (``hx-get /ui/approvals/{id}``). Shows the operator-relevant request detail (op_id / connector_id / proposed_effect / requester principal_sub / created_at) and the Approve + Deny actions. Self-approval invariant (#1401) The **Approve** button is disabled (read-only) when the operator is the requester and the audited break-glass is off (``self_approval_blocked``); its ``aria-label`` names the ``APPROVAL_ALLOW_SELF_APPROVAL`` flag so the operator learns the lever. **Deny** stays enabled (withdrawing your own request is not an escalation). The button state is UX only -- the BFF re-checks server-side and 403s a forged self-approve, surfaced via ``error_*``. CSRF (#1693 / #1754) Each decision is its own ``
`` carrying its OWN ``hx-headers`` ``X-CSRF-Token`` -- HTMX does NOT inherit ``hx-headers`` to child elements, so the token must ride the form that issues the POST, not a wrapping element. The route minted a fresh token for THIS render and re-set the matching ``meho_csrf`` cookie on the same response, so the double-submit pair lines up after the modal swap rotated it. Decided rows render read-only (#1827) When the request has already been decided (``status != "pending"``) the modal shows a decision banner ("Approved by X at T" / "Rejected by X at T" / "Expired at T") and the Approve/Deny actions are NOT rendered -- there is nothing to act on. Only a pending row offers the decision forms. This is what lets the history view's "View" button reuse the same modal for a closed request.
